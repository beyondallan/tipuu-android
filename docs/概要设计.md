# Tipuu 移动端应用概要设计

## 一、项目概述

### 1.1 项目背景

Tipuu 是一款跨平台移动应用，需要集成多种硬件能力（麦克风、扬声器、蓝牙、WiFi、NFC），并与已有服务端API进行交互。应用需支持离线场景下的本地缓存和数据读取。

### 1.2 项目目标

- 跨平台支持：Android + iOS
- 硬件集成：BLE数据传输、音频录制与实时通话、WiFi连接管理
- 离线支持：本地存储能力，支持网络较差环境下的缓存读取
- 易于维护：模块化架构，清晰的代码结构

### 1.3 技术约束

- 已有服务端API接口
- 需支持几百MB的本地缓存
- 大部分数据为只读缓存

## 二、技术选型

### 2.1 开发框架

| 类别 | 选型 | 版本 | 说明 |
|------|------|------|------|
| **开发框架** | Flutter | 3.24+ | 跨平台UI框架 |
| **开发语言** | Dart | 3.5+ | Flutter官方语言 |
| **状态管理** | Riverpod | 2.5+ | 响应式状态管理 |
| **网络请求** | Dio | 5.4+ | HTTP客户端 |
| **本地数据库** | sqflite | 2.3+ | SQLite封装 |
| **键值存储** | shared_preferences | 2.2+ | 轻量KV存储 |
| **安全存储** | flutter_secure_storage | 9.0+ | Token加密存储 |
| **文件存储** | path_provider | 2.1+ | 文件路径管理 |
| **路由** | go_router | 14.0+ | 声明式路由 |
| **蓝牙BLE** | flutter_blue_plus | 1.32+ | BLE通信 |
| **音频处理** | flutter_sound | 9.2+ | 录音/播放 |
| **实时通话** | flutter_webrtc | 0.9+ | WebRTC音频通话 |
| **权限管理** | permission_handler | 11.3+ | 系统权限 |
| **网络状态** | connectivity_plus | 6.0+ | 连接状态监听 |

### 2.2 选型理由

#### 为什么选择Flutter而非原生开发？

1. **跨平台需求明确** - 需同时支持Android和iOS
2. **BLE插件成熟** - flutter_blue_plus是最成熟的跨平台BLE库
3. **音频支持完善** - flutter_sound + flutter_webrtc满足录音和实时通话需求
4. **开发效率高** - 一套代码，双端运行
5. **硬件API可桥接** - 复杂硬件操作可通过Platform Channel调用原生API

## 三、系统架构

### 3.1 整体架构

采用 **Clean Architecture + 模块化设计**，分为三层：

```
┌─────────────────────────────────────────────────────────────┐
│                    Presentation Layer                        │
│                         (展示层)                              │
│  ┌─────────────┐  ┌─────────────┐  ┌─────────────┐          │
│  │    Pages    │  │   Widgets   │  │  Providers  │          │
│  │  (页面)     │  │  (组件)     │  │  (状态)     │          │
│  └──────┬──────┘  └──────┬──────┘  └──────┬──────┘          │
└─────────┼────────────────┼────────────────┼──────────────────┘
          │                │                │
          ▼                ▼                ▼
┌─────────────────────────────────────────────────────────────┐
│                      Domain Layer                            │
│                        (领域层)                               │
│  ┌─────────────┐  ┌─────────────┐  ┌─────────────┐          │
│  │  UseCases   │  │  Entities   │  │ Repositories│          │
│  │  (用例)     │  │  (实体)     │  │  (接口)     │          │
│  └──────┬──────┘  └──────┬──────┘  └──────┬──────┘          │
└─────────┼────────────────┼────────────────┼──────────────────┘
          │                │                │
          ▼                ▼                ▼
┌─────────────────────────────────────────────────────────────┐
│                       Data Layer                             │
│                        (数据层)                               │
│  ┌─────────────┐  ┌─────────────┐  ┌─────────────┐          │
│  │  Services   │  │ DataSources │  │  Models     │          │
│  │  (服务)     │  │ (数据源)    │  │  (模型)     │          │
│  └──────┬──────┘  └──────┬──────┘  └──────┬──────┘          │
└─────────────────────────────────────────────────────────────┘
          │                │                │
          ▼                ▼                ▼
┌─────────────────────────────────────────────────────────────┐
│                    Infrastructure Layer                       │
│                        (基础设施层)                           │
│  ┌─────────────┐  ┌─────────────┐  ┌─────────────┐          │
│  │   Network   │  │   Storage   │  │   Hardware  │          │
│  │  (网络)     │  │  (存储)     │  │  (硬件)     │          │
│  └─────────────┘  └─────────────┘  └─────────────┘          │
└─────────────────────────────────────────────────────────────┘
```

### 3.2 架构原则

| 原则 | 说明 |
|------|------|
| **依赖倒置** | Domain层定义接口，Data层实现，Presentation层调用UseCase |
| **单一职责** | 每个模块/类只负责一个功能 |
| **模块独立** | Feature模块之间无直接依赖，通过Core层通信 |
| **接口抽象** | 硬件服务、存储服务均定义抽象接口，便于替换实现 |

## 五、存储方案

### 5.1 存储架构

采用分层存储策略，根据数据特性选择不同存储方式：

```
┌─────────────────────────────────────────────────────────────┐
│                      存储层级                                │
├─────────────────────────────────────────────────────────────┤
│                                                             │
│  第一层：键值存储 (SharedPreferences)                        │
│  ├── 存储内容：用户信息、Token、偏好设置                     │
│  ├── 数据量：KB级别                                          │
│  └── 特点：快速读写、轻量级                                  │
│                                                             │
│  第二层：结构化存储 (SQLite/sqflite)                         │
│  ├── 存储内容：设备列表、API响应缓存、历史记录               │
│  ├── 数据量：几百MB                                          │
│  ├── 特点：支持查询、索引、结构化                            │
│                                                             │
│  第三层：文件存储 (path_provider)                            │
│  ├── 存储内容：音频文件、下载媒体、日志                      │
│  ├── 数据量：不限                                            │
│  └── 特点：适合大文件、媒体资源                              │
│                                                             │
│  第四层：安全存储 (flutter_secure_storage)                   │
│  ├── 存储内容：敏感Token、密钥                               │
│  ├── 数据量：KB级别                                          │
│  └── 特点：加密存储、系统级安全                              │
│                                                             │
└─────────────────────────────────────────────────────────────┘
```

#### 设备相关
```sql

CREATE TABLE devices (

```

#### 音频相关
```sql

  created_at INTEGER
```

#### API缓存
```sql
-- API响应缓存表
CREATE TABLE api_cache (
  key TEXT PRIMARY KEY,         -- API路径+参数哈希
  data TEXT NOT NULL,           -- JSON响应数据
  created_at INTEGER,
  expires_at INTEGER            -- 过期时间戳

```

### 5.3 缓存策略

| 策略 | 适用场景 | 说明 |
|------|----------|------|
| `cacheFirst` | 设备列表、用户信息 | 优先读缓存，miss则请求网络并缓存 |
| `networkFirst` | 实时数据 | 优先网络，失败读缓存 |
| `staleWhileRevalidate` | 非关键数据 | 先返回缓存，后台更新 |
| `networkOnly` | 登录等操作 | 仅网络请求 |
| `cacheOnly` | 离线模式 | 仅读缓存 |

## 六、硬件模块设计

### 6.1 硬件抽象层

定义统一的硬件管理接口，所有硬件服务遵循相同规范：

```dart
abstract class HardwareManager {
  String get hardwareType;               // 硬件类型标识
  Stream<HardwareState> get state;       // 状态流
  Future<Result<void>> initialize();     // 初始化
  Future<Result<void>> release();        // 释放资源
  Future<bool> isAvailable();            // 是否可用
  bool isReady();                        // 是否就绪
}

enum HardwareState {
  unavailable,   // 不可用
  initializing,  // 初始化中
  ready,         // 就绪
  active,        // 活动中
  error,         // 错误
  releasing,     // 释放中
}
```

### 6.2 BLE蓝牙模块

#### 功能需求
- BLE设备扫描
- 设备连接/断开
- 服务发现
- 数据收发（Characteristic读写）

#### 接口设计
```dart
abstract class BleService implements HardwareManager {
  Stream<BluetoothState> get state;
  Stream<BleDevice> scan({Duration? timeout});
  Future<void> stopScan();
  Future<Result<void>> connect(String deviceId);
  Future<Result<void>> disconnect(String deviceId);
  Future<Result<void>> sendData(deviceId, serviceUuid, characteristicUuid, data);
  Stream<List<int>> receiveData(deviceId, serviceUuid, characteristicUuid);
  Future<Result<List<String>>> discoverServices(String deviceId);
}
```

#### 实现方案
使用 `flutter_blue_plus` 插件，封装为符合架构规范的Service。

### 6.3 音频模块

#### 功能需求
- 音频录制（支持暂停/恢复）
- 音频播放
- 实时语音通话（WebRTC）

#### 接口设计
```dart
// 录音/播放服务
abstract class AudioService implements HardwareManager {
  Stream<RecorderState> get recorderState;
  Future<Result<String>> startRecording({outputPath, sampleRate, channels});
  Future<Result<AudioRecord>> stopRecording();
  Future<Result<void>> pauseRecording();
  Future<Result<void>> resumeRecording();
  Stream<int> get recordingDuration;
  Future<Result<void>> play(String filePath);
  Future<Result<void>> stopPlayback();
}

// 实时通话服务
abstract class AudioCallService implements HardwareManager {
  Stream<AudioCallState> get callState;
  Future<Result<void>> startCall({serverUrl, roomId, token});
  Future<Result<void>> endCall();
  Future<Result<void>> mute();
  Future<Result<void>> unmute();
}
```

#### 实现方案
- 录音/播放：`flutter_sound`
- 实时通话：`flutter_webrtc`，与服务端WebRTC服务对接

### 6.4 WiFi模块

#### 功能需求
- 网络状态监听
- 网络可用性检测

#### 实现方案
使用 `connectivity_plus` 监听网络状态，集成到NetworkInfo服务。

## 八、错误处理机制

### 8.1 错误分类

```dart
// 应用异常分类
AppException
├── NetworkException    // 网络错误（超时、断连）
├── ServerException     // 服务端错误（500、401、404）
├── StorageException    // 存储错误（读写失败）
├── HardwareException   // 硬件错误（蓝牙、音频）
├── PermissionException // 权限错误（未授权）
└── UnknownException    // 未知错误
```

### 8.2 错误流转

```
Data层异常 (DioException, SQLiteException)
         │
         ▼
AppException (统一异常格式)
         │
         ▼
Failure (领域层错误，不含技术细节)
         │
         ▼
UI层展示 (SnackBar、ErrorWidget)
```
